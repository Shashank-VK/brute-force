SECTION 1: PROJECT IDENTITY & STRATEGY
1.1 What Boot Force Is
Boot Force is the official digital hub for a technical club at CMR University. It is a single web application 
that replaces fragmented communication across WhatsApp groups, Google Forms, Google Drives, and 
social media pages. It centralizes member management, event hosting, challenge systems, project 
showcases, media galleries, and performance tracking into one cohesive platform.
It is NOT a multi-tenant platform. It is NOT a club-spawning system. It serves one club — Boot Force — with 
one URL, one database, one deployment.

1.3 Target Users
Visitor (Guest): Anyone who opens the URL without logging in. They can browse every public page — 
homepage, team, events, challenges, projects, gallery, leaderboard, blog. They cannot interact (register, 
submit, like). When they attempt an interactive action, they are prompted to sign in. No login wall. No role 
selection gate.
Student (Member): A registered and authenticated user. They can do everything a visitor can, plus: register 
for events, submit challenge solutions, like projects, view their personal dashboard, earn badges, track 
streaks, write blog posts, download certiﬁcates, and receive in-app notiﬁcations.
Admin: A member with elevated privileges. They can do everything a member can, plus: full CRUD on 
events, challenges, team members, projects, gallery albums. They can review and grade submissions, 
manage users (promote, ban), publish announcements, view platform analytics, and see audit logs. 
Multiple admins are supported. The ﬁrst admin is seeded in the database. Subsequent admins are 
promoted by existing admins.

SECTION 2: SYSTEM ARCHITECTURE
2.1 High-Level Architecture
The application follows a three-layer architecture:
Layer 1 — Client (Browser)
Next.js renders pages. Server Components handle initial data fetching and HTML generation on the server. 
Client Components handle interactivity, real-time subscriptions, and form state in the browser. The 
Supabase browser client manages real-time WebSocket connections.
Layer 2 — Server (Next.js on Vercel)
Next.js API routes and Server Actions handle mutations, ﬁle uploads, and any operation requiring the 
Supabase service role key. Middleware handles session refresh on every request. ISR (Incremental Static 
Regeneration) caches public pages and revalidates at deﬁned intervals.
Layer 3 — Database & Services (Supabase)
PostgreSQL database with Row Level Security. Supabase Auth for authentication. Supabase Storage for 
ﬁles. Supabase Realtime for WebSocket subscriptions.
All three layers communicate via HTTPS. The Supabase anon key is public (safe for browser use with RLS). 
The service role key is server-only (used in Server Actions and API routes, never exposed to the client).
2.2 Data Fetching Architecture
There are exactly four methods of fetching or mutating data. Every page uses one or more of these. No 
exceptions.
Method 1 — Server Component Direct Fetch
Used for: All public pages (homepage, team, events listing, projects, gallery, leaderboard, blog).
How: The Server Component creates a Supabase server client (reads cookies for auth context), queries the 
database directly, and returns rendered HTML. No client-side JavaScript is shipped for the data fetch itself.
When: On initial page load and ISR revalidation.
Method 2 — Server Actions
Used for: All mutations (form submissions, registrations, likes, admin CRUD, grading, ﬁle uploads, proﬁle 
updates).
How: Functions marked with "use server". Called from Client Components. Execute on the server with full 

--- Page 5 ---
access to the service role key if needed. Return typed responses: { success: boolean, data?: T, error?: string 
}.
When: On user action (button click, form submit).
Method 3 — Client-Side Supabase SDK with TanStack Query
Used for: Any Client Component that needs data after initial render — speciﬁcally pages that receive 
server-fetched data as initialData and then manage updates, mutations, and caching client-side.
How: TanStack Query wraps Supabase client queries. initialData and initialDataUpdatedAt are passed from 
Server Components to avoid double-fetching. staleTime is set to match the ISR revalidation window.
When: Client-side navigation, background refetches, after mutations.
Method 4 — API Routes
Used for: Webhook endpoints, any external service callbacks, operations that don't ﬁt the Server Action 
model.
How: Standard Next.js route.ts ﬁles in the app directory.
When: Rarely. Most operations use Server Actions. API routes are reserved for edge cases.
2.3 Server-to-Client Data Handoff Convention
This is a binding convention across the entire application:
1.  Server Component fetches data and captures dataUpdatedAt = Date.now()
2.  Server Component passes both data and dataUpdatedAt as props to the Client Component
3.  Client Component initializes TanStack Query with initialData: data and initialDataUpdatedAt: 
dataUpdatedAt
4.  staleTime is set per query (matching the ISR revalidation window for that page)
5.  On client-side navigation (back/forward, link clicks), TanStack Query compares timestamps. If server 
data is newer, it replaces the cache. If the cache is newer (due to a mutation), the cache wins.
6.  Serialization rule: All data passed from Server to Client Components must be plain objects. No Date 
objects, no undeﬁned values, no class instances. Dates are ISO strings, parsed on the client.
Exception: Real-time components (leaderboard, notiﬁcations, announcement ticker) do NOT use 
initialData. They fetch on mount via TanStack Query, then Supabase Realtime pushes updates. A brief 
skeleton on ﬁrst load is acceptable for these components.

Subscription lifecycle rules:
Each subscription lives in a dedicated custom hook (e.g., useRealtimeLeaderboard)
Subscriptions are created in useEffect with cleanup via supabase.removeChannel() on unmount
Callbacks reference queryClient (stable singleton) to avoid stale closures
Subscriptions are only active on pages that need them — not globally
Connection drop handling:
Supabase Realtime has built-in reconnection with exponential backoff
On CHANNEL_ERROR or TIMED_OUT: show a subtle toast "Live updates paused. Reconnecting..."
On SUBSCRIBED (reconnection success): call invalidateQueries on all real-time queries to catch 
missed updates
No manual polling fallback for the hackathon
Leaderboard update pattern (detailed):

--- Page 8 ---
1.  Realtime event ﬁres with the updated row
2.  Call queryClient.cancelQueries(['leaderboard']) to stop any in-ﬂight fetches
3.  Call queryClient.setQueryData(['leaderboard'], callback) — the callback ﬁnds the user in the cached 
array, updates their score, and re-sorts. If the user isn't in the cached array, skip (the refetch will catch 
it).
4.  Immediately call queryClient.invalidateQueries(['leaderboard']) — this triggers a background refetch 
from the server. When it resolves, it silently replaces the cache with the authoritative sorted list. This 
catches edge cases like users entering the top 100 from outside the cached set.
5.  No setTimeout. No delay. The optimistic setQueryData prevents any loading ﬂash. The 
invalidateQueries corrects silently in the background.

4.2 PostgreSQL Triggers
Trigger 1: Auto-create proﬁle on signup
Fires: AFTER INSERT ON auth.users
Action: Inserts a new row into proﬁles with the user's ID and email-derived name
Purpose: Every auth user always has a corresponding proﬁle row
Trigger 2: Update event registration count
Fires: AFTER INSERT OR DELETE ON event_registrations
Action: Recalculates COUNT(*) from event_registrations where event_id = NEW.event_id (or 
OLD.event_id on delete) and status != 'cancelled', then updates events.registration_count
Purpose: Keeps the denormalized count accurate
Trigger 3: Update project likes count

--- Page 39 ---
Fires: AFTER INSERT OR DELETE ON project_likes
Action: Recalculates COUNT(*) from project_likes where project_id = NEW.project_id, then updates 
projects.likes_count
Purpose: Keeps the denormalized count accurate
Trigger 4: Update user total score
Fires: AFTER INSERT OR UPDATE OF score, time_bonus ON submissions
Action: Recalculates SUM(COALESCE(score, 0) + COALESCE(time_bonus, 0)) from submissions where 
user_id = NEW.user_id and status = 'graded', then updates proﬁles.total_score
Purpose: Leaderboard accuracy. This is the most critical trigger.

SECTION 5: AUTHENTICATION & SECURITY 
ARCHITECTURE
5.1 Auth Flow — Step by Step
Signup (Email/Password):
1.  User ﬁlls signup form: email, password, full name
2.  Client calls Supabase Auth signUp({ email, password, options: { data: { full_name } } })
3.  Supabase creates the user in auth.users
4.  The database trigger ﬁres, creating a proﬁles row with the user's ID and full_name
5.  Default role is 'member'
6.  Session is created, cookies are set via @supabase/ssr
7.  User is redirected to the redirect query parameter URL, or /dashboard if none exists
Signup (OAuth — Phase 7B, if time allows):
1.  Before initiating OAuth, store the redirect URL (from query params) in localStorage
2.  Call supabase.auth.signInWithOAuth({ provider: 'google' }) or 'github'
3.  User is redirected to Google/GitHub, authorizes, returns to callback URL
4.  Supabase handles the token exchange
5.  Database trigger creates the proﬁle row if it's a new user
6.  Middleware refreshes the session
7.  Client reads redirect from localStorage, validates it starts with /, navigates there, clears localStorage
Login:
1.  User enters email and password
2.  Client calls supabase.auth.signInWithPassword({ email, password })
3.  Session cookies set via @supabase/ssr
4.  Redirect to redirect query param or /dashboard
Session Refresh:
Next.js middleware runs on every request. It creates a Supabase server client, calls

supabase.auth.getUser(), which reads and refreshes the session cookies if needed. This ensures Server 
Components always have access to the current session.
Logout:
1.  Client calls supabase.auth.signOut()
2.  Cookies cleared
3.  Redirect to /

6.6 Design Principles
1.  Dark-ﬁrst: The primary experience is dark (navy backgrounds). Light sections are used for contrast and 
variety, not as a separate "mode." There is no light/dark mode toggle. The site is designed as one 
cohesive dark-themed experience with light cards and sections for visual rhythm.
2.  Cinematic composition: Large hero sections with dramatic gradient overlays. Bold typography. 
Generous whitespace. The feeling of a movie poster, not a dashboard.
3.  Sci-ﬁ texture: Subtle glow effects on borders (box-shadow with brand colors at low opacity). 
Monospace font for data elements. Geometric background patterns (subtle CSS-generated, not 
images). The Matrix-style element conﬁned to the loading/splash animation or hero background — 
NOT sitewide.
4.  Mobile-ﬁrst: Every layout designed for mobile ﬁrst, expanded for desktop. Touch targets minimum 
44x44px. No hover-only interactions — everything accessible via tap.
5.  Breathing room: Generous padding (see spacing tokens). Cinematic visuals need room to breathe. 
Cramped layouts kill the aesthetic.
6.  Micro-interactions: Every button has a hover state. Every card lifts on hover. Every interactive 
element gives tactile feedback. But animations serve the interaction, not distract from it.
7.  Consistent card language: All listing pages (events, challenges, projects, team) use the same card 
design tokens. The content varies but the visual container is recognizable.

8.11 Streaks
Login Streak Logic:
On every login (detected in middleware or auth listener):
1.  Read proﬁles.last_active_date
2.  Get today's date in UTC
3.  If last_active_date is yesterday: increment current_streak by 1
4.  If last_active_date is today: do nothing (already counted)
5.  If last_active_date is older than yesterday: reset current_streak to 1
6.  If current_streak > longest_streak: update longest_streak
7.  Set last_active_date to today
All comparisons in UTC dates.
Display:
Dashboard: "Current Streak: X days 🔥 " and "Longest Streak: Y days"
If streak is 7+, show a ﬂame animation next to the number

8B.2 Streak System
Login streak logic (implemented in middleware or auth state change listener):
On each authenticated request where last_active_date != today (UTC):
1.  Read proﬁles.last_active_date
2.  Get today's date in UTC: new Date().toISOString().split('T')[0]
3.  Get yesterday's date in UTC
4.  If last_active_date === yesterday: increment current_streak by 1
5.  If last_active_date === today: do nothing (already counted today)
6.  If last_active_date is older than yesterday OR null: reset current_streak to 1
7.  If current_streak > longest_streak: update longest_streak = current_streak
8.  Set last_active_date = today
9.  Single UPDATE query to proﬁles
Important: this runs in middleware on the server, NOT in the browser. Prevents manipulation.
Performance: one additional DB query per ﬁrst-request-of-the-day. Negligible.
Streak display on dashboard:
"Current Streak: X days" with a ﬁre emoji/icon
"Longest Streak: Y days"
If current streak >= 7: animated ﬂame icon (Framer Motion pulse animation)
If current streak >= 30: special "On Fire" visual treatment

Upload Flow (single ﬁle):
1.  Client validates ﬁle type and size
2.  Client calls Server Action with the ﬁle
3.  Server Action uploads to Supabase Storage with a unique path: 
{bucket}/{entityId}/{timestamp}_{ﬁlename}
4.  If upload succeeds, server updates the database row with the new public URL
5.  If the entity previously had an image (replacement), server deletes the old ﬁle from Storage
6.  If old ﬁle deletion fails, log the orphan path to audit_log (non-blocking)
7.  Return success with the new URL
Upload Flow (bulk — gallery):
1.  Client validates each ﬁle individually (type, size)
2.  Invalid ﬁles are rejected with per-ﬁle error messages before upload begins
3.  Valid ﬁles are uploaded in parallel via Promise.allSettled()
4.  Progress: track fulﬁlled count vs total. Display "Uploaded 7 of 12"
5.  After all settle: collect fulﬁlled URLs, report rejected ﬁles with reasons
6.  Batch insert fulﬁlled URLs into gallery_images via single .insert([...])
7.  If batch insert fails: retry once
8.  If retry fails: delete all successfully uploaded ﬁles from Storage, show error "Upload failed. Please try 
again."
9.  Result: either all succeed (ﬁles in Storage AND database) or all are cleaned up (nothing orphaned)

APPENDIX G: NOTIFICATION QUERY PATTERN
The notiﬁcation bell combines two data sources into one list:
Source 1 — User Notiﬁcations:
sql
SELECT id, title, message, type, is_read, link, created_at, 'notiﬁcation' as source
FROM notiﬁcations 
WHERE user_id = \$1 
ORDER BY created_at DESC 

--- Page 143 ---
LIMIT 20
Source 2 — Active Announcements (not dismissed):
sql
SELECT id, title, content as message, 'announcement' as type, false as is_read, null as link, created_at, 
'announcement' as source
FROM announcements 
WHERE is_active = true 
AND created_at > (SELECT last_dismissed_announcements_at FROM proﬁles WHERE id = \$1)
ORDER BY created_at DESC
Combined: Both queries run, results merged on the client, sorted by created_at descending. 
Announcements appear interleaved with notiﬁcations in chronological order.
Marking as read:
Notiﬁcations: UPDATE notiﬁcations SET is_read = true WHERE id = \$1 AND user_id = \$2
Announcements: "Dismiss all" updates proﬁles.last_dismissed_announcements_at = now()
Unread count: Count of notiﬁcations where is_read = false PLUS count of undismissed active 
announcements.
